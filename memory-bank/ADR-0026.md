# ADR-0026: Use .env as single source of truth for component versions

## Context
Version numbers were scattered across multiple files: `extension/package.json`, `mcp/package.json`, CHANGELOG.md, MEMORY.md, and the sidebar view container title. The `sync-versions.js` script treated `package.json` files as the source of truth and propagated to other files. This led to version drift — notably CHANGELOG.md listing "Extension v0.7.0" while `extension/package.json` still had `0.6.6`, because CHANGELOG versions were written by hand with no validation against the canonical source.

## Decision
A `.env` file at the repository root serves as the single source of truth for all component versions:

```env
EXT_VERSION=0.7.0
MCP_VERSION=2.4.0
```

`scripts/sync-versions.js` reads from `.env` and propagates to all downstream files:
- `extension/package.json` (version field + `viewsContainers.activitybar[].title`)
- `mcp/package.json` (version field)
- `extension/mcp-server/package.json` (bundled copy)
- MEMORY.md (section headers)

The `--bump`, `--set-ext`, `--set-mcp` flags update `.env` first, then propagate. The `--sync` flag reads `.env` and writes to all targets. Report mode (no args) shows `.env` values and flags any drift in downstream files.

`.env` is tracked in git (not gitignored) since it IS the source of truth.

## Alternatives Considered

### Keep package.json as source of truth
The previous approach. Simple but caused drift because CHANGELOG and other docs had no validation against the canonical version. Two separate sources (ext + mcp package.json) made it easy to update one and forget the other.

### Use a versions.json file
A dedicated JSON file for versions. Functionally equivalent to .env but more complex to parse in shell scripts and less conventional for configuration values.

## Consequences
- Single place to check and update versions — reduces drift risk
- .env is the canonical source; package.json files are derived (updated by sync script)
- All existing build scripts (bump-version.sh, auto-version.js, npm run build:ext) work unchanged — they delegate to sync-versions.js
- .env must be committed to git (not gitignored) since it contains project metadata, not secrets
- CHANGELOG.md remains append-only — versions in CHANGELOG headers are snapshots, not updated by sync
