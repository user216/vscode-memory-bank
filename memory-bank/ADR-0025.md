# ADR-0025: Remove NOTE Type — Redundant in Knowledge Ecosystem

**Status:** Accepted
**Date:** 2026-04-03
**Deciders:** Project maintainer

## Context

The memory bank supports 5 item types: projectbrief (core), TASK, ADR (decision), NOTE, and README (structure). The NOTE type was introduced as a "general-purpose knowledge item" for patterns, insights, and reference material.

In practice, notes have never been used — zero NOTE items exist in this project's memory bank. More importantly, the broader tooling ecosystem already provides better-fit mechanisms for every use case that notes were meant to serve:

1. **Skills** (SKILL.md) — reusable workflows, recipes, step-by-step procedures
2. **Instructions** (.instructions.md) — conventions, patterns, coding guidelines, project rules
3. **AI auto-memory** (MEMORY.md / CLAUDE.md) — agent-learned patterns, preferences, gotchas
4. **ADR decisions** — any insight important enough to persist ("we use X because Y")
5. **TASK progress logs** — ephemeral context tied to ongoing work

This leaves notes in an identity crisis: they have no clear lifecycle (no status progression like tasks, no accept/reject like decisions), no natural expiry, and they compete for token budget in `memory_recall` against more actionable items. Users and agents face unnecessary friction deciding "is this a note or a decision?" when the answer is almost always "decision" or "doesn't belong in memory bank at all."

## Decision

Hard-remove the NOTE type from the memory bank data model. This is a full removal, not a deprecation.

**Changes:**
- Remove `memory_create_note` MCP tool entirely (delete `mcp/src/tools/memory-create-note.ts`)
- Remove `"note"` from `ItemType` union type
- Remove NOTE from `memory_schema`, `memory_query` type filter, `memory_search` type filter
- Remove Notes sidebar view from VS Code extension (delete `NotesTreeProvider`)
- Update instructions, skills, agents, prompts to stop mentioning notes
- Change `deriveType()` fallback from `"note"` to `"structure"` — unrecognized files are structural
- Remove note tier from `memory_recall` active priority ordering

**No migration needed** — zero notes exist in practice. All knowledge storage is covered by:
- **projectbrief** — what the project is
- **TASK** — what we're doing (with progress logs)
- **ADR** — why we decided X
- **Skills** — reusable workflows
- **Instructions** — conventions and patterns
- **AI auto-memory** — agent-learned insights

## Alternatives Considered

### Keep notes as-is
Notes remain available for general knowledge. Rejected because they overlap with instructions, skills, and AI auto-memory — zero adoption confirms lack of unique value.

### Merge notes into ADR type with a 'Reference' status
Add a new ADR status for non-decision reference material. Rejected because it muddies the ADR concept — decisions should be decisions, not a catch-all.

### Deprecate first, remove later
Soft-deprecate the type and remove in a future version. Rejected because zero notes exist in practice, so there is nothing to migrate and no user impact from immediate removal.

## Consequences

- Simpler mental model — users and agents only choose between TASK and ADR
- Reduced token waste in memory_recall — no notes competing for budget
- Cleaner sidebar — 3 views instead of 4
- Knowledge storage responsibility is clearer: instructions for conventions, skills for workflows, ADRs for decisions, auto-memory for agent learnings
- Breaking change for anyone who built workflows around `memory_create_note`
- Existing NOTE-NNN files are classified as `"structure"` type (still indexed and searchable)
- `ItemType` union reduced to 4 values: `core`, `task`, `decision`, `structure`
