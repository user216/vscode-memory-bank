# ADR-0023: Enhanced sidebar — titles, tags, sort, relations

## Context
The VS Code extension sidebar showed only file IDs (e.g., `TASK-001`) as labels and status as description. Tags from YAML frontmatter were invisible in Tasks and Decisions views (only Notes showed tags). The `title:` field from frontmatter was unused. Relations between items (from `related:` frontmatter and cross-references in body text) were not surfaced in the tree views. Additionally, `extractStatus()` and `extractTags()` were duplicated across providers.

## Decision
Enhance all three sidebar tree providers (Tasks, Decisions, Notes) with:

1. **Shared parsing utilities** (`frontmatter-utils.ts`) — consolidate duplicated `extractStatus()`, `extractTags()`, add `extractTitle()`, `buildRelations()`, `buildDescription()`, and a shared `RelationItem` class.

2. **Titles as labels** — show the `title:` from YAML frontmatter (or H1 heading with ID prefix stripped) as the tree item label. Fall back to file ID if no title.

3. **Tags in description** — show tags alongside status in the description field, separated by `·` (middle dot).

4. **Status-based sorting** — Tasks sort by: In Progress → Pending → Completed → Abandoned. Decisions sort by: Proposed → Accepted → Deprecated → Superseded → Rejected.

5. **Expandable relations** — items with relations (from `related:` frontmatter, cross-references like `TASK-NNN`/`ADR-NNNN`/`NOTE-NNN` in body text, and `[[wikilinks]]`) become expandable tree nodes. Child `RelationItem` nodes show the target ID with relation type as description and a purple link icon. Clicking opens the target file.

## Alternatives Considered

### Show tags only (no titles, no sort, no relations)
Simplest change — just add tag extraction to Tasks and Decisions providers. Rejected because showing the file ID alone as a label is unhelpful when tasks have descriptive titles.

### Separate "Relations" tree view
A dedicated view panel for all cross-references. Rejected because relations are most useful in context — seeing what a specific task references is more valuable than a flat list of all relations.

### Read relations from MCP server graph
The MCP server maintains an in-memory link graph with richer data (user-created links via `memory_link`). Rejected because the extension and MCP server run in separate processes — the extension can only read files from disk. File-derived relations (frontmatter `related:` + cross-refs + wikilinks) cover the persistent cases.

## Consequences
- Tree items with titles are more scannable than bare IDs like `TASK-001`
- Tags visible at a glance without opening each file
- Status-based sort puts actionable items (In Progress, Proposed) at the top
- Relations provide navigable links between items directly in the sidebar
- Shared `frontmatter-utils.ts` eliminates code duplication across 3 providers
- `TreeDataProvider` type signatures now include `RelationItem` union type
- Items without relations remain flat (no expansion arrow)
