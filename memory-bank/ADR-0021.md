# ADR-0021: 3-Layer ADR Compliance Verification System

## Context

After implementing ADR-0015 §7 (eliminate v1 core files), we discovered that accepted ADRs had implementation gaps that went undetected — the code continued to create, reference, and prioritize files that the ADR explicitly marked for elimination. This class of issue (accepted-but-not-implemented ADRs) is systemic and requires automated enforcement rather than manual review.

## Decision

Implement a 3-layer verification system that checks ADR compliance at different points:

### Layer 1: MCP Tool (`memory_verify_decisions`)

A new MCP tool that parses machine-readable assertions embedded in ADR `## Verification` sections and executes them against the project filesystem. Assertions use HTML comments so they are invisible when rendered:

```markdown
### Verification
<!-- ASSERT_FILE_EXISTS: path/to/expected/file -->
<!-- ASSERT_FILE_NOT_EXISTS: path/to/eliminated/file -->
<!-- ASSERT_CONTAINS: path/to/file | expected string -->
<!-- ASSERT_NOT_CONTAINS: path/to/file | forbidden string -->
```

The assertion engine (`mcp/src/verify-assertions.ts`) is self-contained — it works without the MCP store by reading files directly with gray-matter, making it reusable by tests and hooks.

### Layer 2: Vitest Compliance Test

`mcp/test/adr-compliance.test.ts` runs against the real project codebase (not fixtures) and verifies that all accepted ADRs with assertions pass. This runs in CI via `npm test`.

### Layer 3: Session-Start Hook

`hooks/scripts/session-start.sh` extracts assertions from accepted ADRs using grep and runs bash-level checks. Prints warnings for failures at session start so issues are caught proactively.

## Alternatives Considered

### Manual review only
- Rejected: This is exactly what failed — ADR-0015 §7 was accepted but not implemented for months.

### CI tests only (Layer 2 alone)
- Rejected: Tests only run when code changes are committed. Doesn't catch drift during development sessions.

### MCP tool only (Layer 1 alone)
- Rejected: On-demand tool requires the developer to remember to run it. No proactive warning.

## Consequences

- Every accepted ADR can now include machine-checkable assertions
- ADR-0021 itself is self-referential — its assertions verify the verification system
- New ADRs should include a `## Verification` section as a best practice
- Session-start hook provides early warning without requiring any manual action
- The assertion engine has zero store dependency, making it portable
